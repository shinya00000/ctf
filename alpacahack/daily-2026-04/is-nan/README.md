# is-nan writeup

## 1. 概要
16文字の16進数文字列を入力し，それが特定の条件を満たせばフラグが得られる問題です．条件とは，入力文字列が `"deadbeef"` を含み，かつIEEE 754形式の倍精度浮動小数点数（`double`）として解釈した際に `NaN`（非数）となることです．

## 2. 問題の分析

### ファイル構成
配布ファイルには，C言語のソースコード `chal.c`，コンパイル済みの実行バイナリ `chal`，および環境構築用の `Dockerfile` や `compose.yaml` などが含まれています．サーバ上では指定されたポートで `chal` が稼働しています．

### ソースコードの確認
`chal.c` では，標準入力から文字列を受け取り，以下の検証を行っています．
1. 文字列長が正確に16文字であること．
2. すべての文字が16進数として有効であること．
3. 文字列中に `"deadbeef"` が含まれていること．
4. 入力された16進数文字列を `uint64_t` 型としてパースし，それを `double` 型の変数にコピーした際，`isnan()` 関数が真を返すこと．
これらをすべてクリアした場合のみ，`flag.txt` の内容が出力されます．

## 3. 脆弱性と攻略方針

### 脆弱性
プログラム自体にメモリ破壊などの脆弱性は存在しませんが，IEEE 754における浮動小数点数の内部表現を利用したパズルとなっています．意図的に `NaN` と判定されるビット列を構築し，その中に指定された文字列を埋め込むことが可能です．

### 攻略方針：
64ビットの倍精度浮動小数点数（`double`）において，`NaN` となる条件は「指数部（11ビット）がすべて `1` であり，かつ仮数部（52ビット）が `0` でないこと」です．
これを16進数16桁の入力に当てはめると，上位12ビット（先頭の3桁）で符号ビットと指数部を決定できます．指数部をすべて `1` にするためには，先頭の3桁を `7ff`（符号部0）または `fff`（符号部1）にする必要があります．
残りの13桁で仮数部を構成しますが，これが `0` 以外であれば `NaN` として成立します．したがって，この仮数部の領域に必須条件である `"deadbeef"` を配置し，16桁の長さに合わせるため残りを任意の16進数（たとえば `f`）で埋めればよいことになります．
これにより，`ffffffffdeadbeef` などの文字列がすべての条件を満たすペイロードとなります．
