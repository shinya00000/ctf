# fake-shuffle-proof　writeup
<https://alpacahack.com/daily/challenges/fake-shuffle-proof>

## 1. 概要

本問題は提供されたPythonスクリプトにおいて，カードのペアに対するハッシュ値（proof）の発行と検証のロジックに存在する脆弱性を突く問題です．特定の条件を満たすカードの組み合わせを用いて，異なる入力から同じハッシュ値を生成させることでフラグを取得します．

## 2. 問題の分析

### ファイル構成

server.py

### ソースコードの確認

カードは0から3までの数字で管理され，それぞれスートとランク（A♠，A♥，K♠，K♥）が割り当てられています．
機能としてissueとverifyが存在します．
issue機能では異なる色（スート）のカードのペアを送信すると，そのランク情報に基づくハッシュ値が生成・出力されます．
verify機能では同じ色のカードのペア（ただし異なるカード）とそのペアに対して期待されるハッシュ値を送信することでフラグが得られます．
ハッシュ化を行うrank_only_commit関数は入力されたカード番号を2で割り切り捨てた値（ランク）をソートしてハッシュ化しています．

## 3. 脆弱性と攻略方針

### 脆弱性

rank_only_commit関数においてカードの色情報が計算過程で切り捨てられる点，および引数の順序がsorted()により無視される点に脆弱性があります．これにより，異なるカードのペアであっても，ランクの組み合わせが同じであれば完全に同一のハッシュ値が生成されます．

### 攻略方針：

フラグを取得するためにはverifyで「同じ色で異なるカード」のペア（例えば0と2）を入力し，検証を通過する必要があります．このペアに対するrank_only_commitの結果（ランクは0と1）を，事前にissueを用いて取得します．
issueでは「異なる色」である必要があるため，例えば0（A♠，黒，ランク0）と3（K♥，赤，ランク1）のペアを入力します．これにより，0と2のペアに対して要求されるものと同じハッシュ値を取得できるため，これをそのままverifyで提出して制限を突破します．
