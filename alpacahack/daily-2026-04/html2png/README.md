# html2png Writeup
<https://alpacahack.com/daily/challenges/html2png>

## 1. 概要

ユーザーが入力したHTMLをサーバーサイドのPuppeteer（Chromium）でレンダリングし，スクリーンショットをPNG画像として返すWebアプリケーションである．
フラグはサーバーのルートディレクトリに配置されているが，ファイル名にランダムなハッシュ値が含まれているため事前には推測できない仕様となっている．また，問題文のヒントからRCE（リモートコード実行）は不要であることがわかっている．

## 2. 問題の分析

### ファイル構成
Node.jsとExpressを用いたWebサーバーであり，主なロジックは `web/index.js` に記述されている．Docker環境で動作しており，フラグは起動時に `web/flag.txt` からハッシュ付きのファイル名（`/flag-<hash>.txt`）に変更されて配置される．

### ソースコードの確認
`index.js` の処理を確認すると，ユーザーがPOST送信したHTML文字列を一度 `/tmp/<ランダムUUID>.html` として一時ファイルに保存している．その後，Puppeteerが `page.goto("file:///tmp/<ランダムUUID>.html")` を実行し，保存したローカルファイルを開いてスクリーンショットを撮影する処理となっている．
Chromiumが初めから `file://` スキームのコンテキストで動作しているため，HTML内のタグを用いて同一サーバー内の他のローカルファイルを容易に参照できる状態となっている．

## 3. 脆弱性と攻略方針

### 脆弱性
サーバー側でユーザーの入力をローカルのHTMLファイルとして保存し，それをヘッドレスブラウザで直接開いて画像化している点である．ブラウザがローカルファイルコンテキストで動作するため，HTML内に `file:///` を参照する要素を埋め込むことで，同一オリジンポリシー等の制約を気にすることなく，ディレクトリリスティングや任意のファイルの内容をレンダリング結果（スクリーンショット画像）として出力させることが可能となる．

### 攻略方針：
1. ルートディレクトリ（`/`）の内容を `<iframe>` 等で表示させるHTMLを送信し，レンダリングされた画像からフラグの正確なファイル名（`flag-<hash>.txt`）を特定する．
2. 特定したファイルパスを指定した `<iframe>` を含むHTMLを再度送信し，得られた画像からフラグの文字列を直接読み取る．

## 4. エクスプロイトコード

ファイル名を特定するためのペイロード：
```html
<iframe src="file:///" width="100%" height="800px"></iframe>
```
上記を送信すると，ルートディレクトリのファイル一覧がレンダリングされた画像が返却され，`flag-3f1816a5.txt` というファイルが存在することが確認できた．

フラグを取得するためのペイロード：
```html
<iframe src="file:///flag-3f1816a5.txt" width="100%" height="800px"></iframe>
```
上記を送信して得られた画像にフラグが描画されており，フラグが得られた．
