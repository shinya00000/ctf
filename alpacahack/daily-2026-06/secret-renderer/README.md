# secret-renderer writeup
<https://alpacahack.com/daily/challenges/secret-renderer>

## 1. 概要

本問題はNginxとFlask間のURL解釈の差異を突いたWeb Cache Deceptionの脆弱性を利用し管理者のセッション情報をキャッシュさせてフラグを奪取するWeb問題である．

## 2. 問題の分析

### ファイル構成

NginxによるリバースプロキシとバックエンドのFlaskアプリケーションから構成されている．

### ソースコードの確認

`nginx/default.conf`では拡張子が`.css`や`.js`であるリクエストに対してプロキシキャッシュを有効にし7日間保持する設定になっている．
一方`backend/app.py`では`/render/<lang>`というルーティングが存在し`<lang>`に指定された言語が存在しない場合はデフォルトの英語テンプレートを使用してユーザー固有のシークレットを描画する．

## 3. 脆弱性と攻略方針

### 脆弱性

Web Cache Deceptionが発生する．NginxはリクエストされたURLの末尾を見て静的ファイルであると判定しキャッシュを行う．しかしバックエンドのFlaskは指定された拡張子付きのURLを単なるパラメータとして解釈し動的な個人情報を含むページを正常に返す．NginxとFlaskでファイルとパラメータの解釈にズレが生じている．

### 攻略方針：

ブラウザの機能のみを用いて以下の手順で攻撃を行う．

1. 報告機能（`/report`）において管理者のbotに`http://nginx:3000/render/en.css`というURLへアクセスさせる．
2. 管理者がこのURLにアクセスするとバックエンドのFlaskが管理者のフラグを含んだHTMLを生成して返す．
3. NginxはこのレスポンスをCSSファイルへのアクセスであると誤認し管理者のフラグが含まれたページをそのままキャッシュサーバーに保存する．
4. 直後に攻撃者が自身のブラウザで`http://(address):(port)/render/en.css`にアクセスする．
5. Nginxはバックエンドへリクエストを送る代わりに保存済みのキャッシュを攻撃者にそのまま返すことでフラグを読み取ることができる．

これにより画面上にキャッシュされた管理者のページが表示されフラグを取得できる．
